# Uni-NaVid 模块级改进方向

> 从大模块视角出发，聚焦可替换、可升级的核心组件。
> 按 **难度 × 预期效果** 综合排序，性价比高的改进优先。
> 生成日期：2026-05-26

---

## 排序说明

```
难度：  低（数小时内完成）| 中（1-3天）| 高（1-2周）| 极高（需仿真环境/大量算力）
效果：  低（边际提升）   | 中（可见提升）| 高（显著提升）| 极高（质的飞跃）
```

综合优先级 = 效果 ÷ 难度，效果相同时难度低者优先。

## 第一梯队：低难度 × 高效果（立刻可做）

> 改动小，收益大，应当优先实施。

---

### 1. 语言模型替换：Vicuna-7B → LLaMA-3.1-8B

> ⚠️ **当前执行计划已选定改进 7（Qwen2.5-7B）作为语言模型替换方案**，原因是后续任务可能包含中文指令。改进 1 保留作为备选参考，若切换为纯英文任务场景可重新评估。

| 项目       | 当前              | 替换后              |
| -------- | --------------- | ---------------- |
| 基座       | LLaMA-2（2023年中） | LLaMA-3.1（2024年） |
| 上下文      | 4K token        | 128K token       |
| MT-Bench | 6.57            | 8.22             |
| 架构兼容性    | —               | 高度兼容，改动极小        |


**为什么难度低**：LLaMA-3.1 与 LLaMA-2 架构高度兼容，`llava_llama_vid.py` 几乎不需要修改，只需更换权重路径并升级 `transformers ≥ 4.40`。

**预期效果**：指令跟随、复杂推理、长上下文理解全面提升；128K 上下文可支持更长导航历史，无需额外压缩。

```
替换步骤：
  1. 下载 meta-llama/Meta-Llama-3.1-8B-Instruct
  2. 修改 scripts/uninavid_stage_1.sh 中 PREV_MODEL 路径
  3. pip install "transformers>=4.40"
  4. 用新基座重新运行 Stage 1 训练
```

---

### 2. 训练范式升级：SFT → SFT + DPO


| 项目   | 当前                  | 升级后                   |
| ---- | ------------------- | --------------------- |
| 范式   | SFT（CE Loss）        | SFT 预热 + DPO 对齐       |
| 监督信号 | 仅正确动作               | 成功轨迹 vs 失败轨迹对比        |
| 工具依赖 | HuggingFace Trainer | HuggingFace TRL（原生支持） |


**为什么难度低**：TRL 库原生提供 `DPOTrainer`，数据格式为 `{"prompt", "chosen", "rejected"}` 三元组，无需修改模型结构。

**预期效果**：模型从"知道正确答案"升级为"知道为什么这个更好"，导航成功率（SR）可提升 3-8%（参考 NaVRL 等工作）。

```
数据构造：
  同一起点出发，记录：
    chosen:  成功到达目标的动作序列
    rejected: 走错路/超时失败的动作序列
  可从已有 VLN-CE 仿真器日志中直接提取
```

## 第二梯队：低难度 × 中效果（顺手可做）

> 改动量小，各有一定收益，可与其他工作并行推进。

---

### 3. 推理量化：bf16 → AWQ 4-bit


| 项目      | 当前（bf16） | 量化后（4-bit）      |
| ------- | -------- | --------------- |
| 显存需求    | ~16 GB   | ~8 GB           |
| 精度损失    | —        | < 1%            |
| 可运行 GPU | A100/A40 | RTX 3090 / 4080 |
| 推理速度    | 基准       | 快约 1.5×         |


**为什么难度低**：仅需在 `builder.py` 的 `load_pretrained_model()` 中增加量化加载分支，2-3 行代码。

**预期效果**：部署门槛大幅降低，可在消费级 GPU 上运行；适合快速验证和演示。

```python
# 接入方式（在 builder.py 中）
from awq import AutoAWQForCausalLM
model = AutoAWQForCausalLM.from_pretrained(model_path, load_in_4bit=True)
```

### 4. 动作空间扩展：4 离散 → 细粒度离散


| 当前动作    | 扩展后动作                                     |
| ------- | ----------------------------------------- |
| forward | forward_fast / forward_slow               |
| left    | left_15 / left_30 / left_45 / left_90     |
| right   | right_15 / right_30 / right_45 / right_90 |
| stop    | stop                                      |


**为什么难度低**：LLM 不需要改动，仅需修改训练标注格式和 `offline_eval_uninavid.py` 中的动作解析与轨迹更新逻辑。

**预期效果**：转弯精度提升，尤其在需要小角度调整的狭窄走廊、门口场景中，路径更顺滑。

---

### 5. 轻量语言模型替换：Vicuna-7B → Phi-3.5-mini（3.8B）


| 项目       | Vicuna-7B | Phi-3.5-mini |
| -------- | --------- | ------------ |
| 参数量      | 7B        | 3.8B         |
| 推理速度     | 基准        | 约快 2×        |
| MT-Bench | 6.57      | 8.38         |
| 上下文      | 4K        | 128K         |


**为什么难度低**：Phi-3 系列基于 Llama 架构变体，继承关系改动小。

**预期效果**：在追求推理速度或资源受限场景下（边缘设备、实机部署），推理频率可从 5 Hz 提升至接近 10 Hz。

---

### 6. 训练策略升级：随机采样 → 课程学习

**当前问题**：`LazySupervisedDataset` 随机采样，简单和困难样本混合训练，初期收敛慢。

**改进方案**：按路径难度（步数、转弯次数、指令复杂度）从易到难排列训练样本。

**为什么难度低**：改动仅在 `LazySupervisedDataset` 的采样策略，无需修改模型结构。

**预期效果**：训练收敛速度加快约 20-30%，最终精度与随机采样持平或略优。

## 第三梯队：中难度 × 高效果（近期规划）

> 需要 1-3 天工程实现，但收益显著，值得投入。

### 7. 语言模型替换：Vicuna-7B → Qwen2.5-7B（中文场景）✅ 已选定

> ✅ **当前执行计划的模块 A 已选定本方案**（取代改进 1）。后续任务可能使用中文指令，Qwen2.5-7B 的中文理解能力是决定性优势。对应分支：`feat/qwen25`，对应 checkpoint：`uninavid_s1_qwen25_curriculum`。

| 项目             | Vicuna-7B        | Qwen2.5-7B       |
| -------------- | ---------------- | ---------------- |
| 中文理解           | 弱                | 极强               |
| MT-Bench       | 6.57             | 8.40             |
| AlignBench（中文） | 落后               | 领先               |
| 上下文长度          | 4K               | 128K             |
| 架构             | LlamaForCausalLM | Qwen2ForCausalLM |


**为什么难度中**：需将 `llava_llama_vid.py` 的继承从 `LlamaForCausalLM` 改为 `Qwen2ForCausalLM`，并适配对话模板格式，约 1 天工作量。

**预期效果**：导航指令含中文时理解准确率大幅提升；Qwen2.5 的推理能力（MT-Bench 8.40）不弱于 LLaMA-3.1-8B（8.22），在中英文混合场景中综合表现更优。

```
替换步骤：
  1. 下载 Qwen/Qwen2.5-7B-Instruct
  2. 修改 llava_llama_vid.py 继承关系（LlamaForCausalLM → Qwen2ForCausalLM）
  3. 在 conversation.py 中增加 Qwen ChatML 对话模板
  4. 检查 tokenizer 特殊 token（<|im_start|> / <|im_end|>）兼容性
  5. pip install "transformers>=4.37"
  6. 修改 scripts/uninavid_stage_1.sh 中 PREV_MODEL 路径
  7. 用新基座重新运行 Stage 1 训练
```

### 8. 视觉编码器替换：EVA ViT-g → SigLIP-SO400M


| 项目   | EVA ViT-g | SigLIP-SO400M    |
| ---- | --------- | ---------------- |
| 参数量  | ~1B       | ~400M            |
| 输出维度 | 1408      | 1152             |
| 权重大小 | ~1.8 GB   | ~0.8 GB          |
| 图文对齐 | InfoNCE   | Sigmoid Loss（更强） |
| 发布时间 | 2022      | 2023（ICCV）       |


**为什么难度中**：需新增 `siglip_encoder.py`，修改 `multimodal_encoder/builder.py` 路由，调整 `mm_projector` 输入维度（1408→1152），并重新训练 projector（视觉编码器 frozen，约 1-2 小时）。

**预期效果**：图文对齐质量提升，视觉问答和指令理解更准确；模型更轻，推理更快。

```
视觉编码器维度变化：
  EVA ViT-g (1408) → SigLIP (1152)
  ↓
  mm_projector: 1408→4096  改为  1152→4096
  ↓
  config.mm_hidden_size = 1152
```

---

### 9. 动作输出扩展：文本解析 → 连续动作回归

**改进方案**：在 LLM 的 `lm_head` 之后增加轻量 MLP 回归头，直接输出连续值 `(v, ω)`（线速度、角速度），损失函数改为 Huber Loss。

**为什么难度中**：需修改 `llava_llama_vid.py` 的 forward 输出结构，以及 `train.py` 的 loss 计算逻辑，约 1-2 天。

**预期效果**：动作更平滑连续，避免离散动作的抖动；适合实机部署和 ROS 接口对接，导航路径更自然。

## 第四梯队：中难度 × 中效果（按需实施）

> 有一定工程量，但收益相对局限，可按研究需要选择实施。

---

### 10. Token 压缩替换：固定阈值 → ToMe（无需训练）

**当前问题**：`process_tensor` 使用固定余弦相似度阈值 0.985，对静态场景过度合并、对动态场景合并不足。

**改进方案**：引入 Token Merging（ToMe），在 EVA ViT 的 Transformer Block 内部用双向软匹配合并 token，**无需重新训练**。

**为什么难度中**：需修改 `eva_vit.py` 的 Transformer Block，在注意力计算前后插入 ToMe merge/unmerge 步骤，约 1 天。

**预期效果**：吞吐量提升约 2×，精度下降 < 0.5%；同时消除固定阈值带来的场景不适应问题。

**参考**：[Token Merging: Your ViT But Faster](https://arxiv.org/abs/2210.09461)，ICLR 2023。

---

### 11. 推理框架替换：HF generate → vLLM

**改进方案**：将 `offline_eval_uninavid.py` 中的 `model.generate()` 替换为 vLLM 的 `LLM.generate()` 接口，利用 PagedAttention 提升批量评测吞吐。

**为什么难度中**：vLLM 对自定义多模态模型的支持需要额外适配（注册视觉 encoder 处理逻辑），约 1-2 天。

**预期效果**：批量评测时吞吐提升 3-5×，适合大规模 benchmark 跑分场景；单步推理延迟改善有限。

---

### 12. Token 压缩替换：固定阈值 → 可学习时序卷积

**改进方案**：参考 VideoLLaMA2，用可学习的时序卷积（Temporal Conv）替代固定余弦相似度合并，端到端训练帧重要性。

**为什么难度中**：需新增时序卷积模块并集成进 `vlm_attention`，以及对应的训练 loss，约 2-3 天。

**预期效果**：模型自动学习哪些帧重要，动态场景（快速移动、目标切换）理解更准确。

**参考**：[VideoLLaMA 2](https://arxiv.org/abs/2406.07476)，arXiv 2024。

---

### 13. 视觉编码器补充：DINOv2-ViT-L（空间定位优先）

**改进方案**：若导航的瓶颈是空间定位（而非图文对齐），可引入 DINOv2-ViT-L 的空间特征，与 SigLIP 做双编码器融合。

**为什么难度中**：需新增编码器并设计双编码器融合策略（concat / cross-attn），约 2 天。

**预期效果**：深度感知、物体位置判断、场景结构理解提升，对 ObjectNav 任务尤为有效。

**参考**：[DINOv2](https://arxiv.org/abs/2304.07193)，TMLR 2024。

## 第五梯队：高难度 × 高效果（长期规划）

> 需要较长工程周期，但对模型能力有结构性提升，适合作为长期研究方向。

---

### 14. 多模态投影头升级：MLP → Resampler

**改进方案**：引入 Flamingo 式 Resampler——固定数量的可学习 latent（如 64 个），通过交叉注意力"读取"任意数量的视觉特征，输出固定 token 数。

**为什么难度高**：Resampler 模块本身需要设计与实现，且需要与现有 `token_generation` / `process_tensor` 机制整合或替代，约 1-2 周。

**预期效果**：彻底解决长视频 token 爆炸问题，无论导航多少步，输入 LLM 的视觉 token 数量始终固定；大幅降低显存压力。

**参考**：[Flamingo](https://arxiv.org/abs/2204.14198)，NeurIPS 2022；[Otter](https://arxiv.org/abs/2305.03726)。

---

### 15. 视觉编码器升级：EVA ViT-g → InternViT-6B


| 项目         | EVA ViT-g | InternViT-6B |
| ---------- | --------- | ------------ |
| 参数量        | ~1B       | ~6B          |
| OCR 与细粒度理解 | 一般        | 极强           |
| 空间关系推理     | 一般        | 强            |
| 显存占用       | 中         | 高            |


**为什么难度高**：InternViT-6B 输出维度 3200，projector 结构需重新设计；显存需求大，训练配置需要大幅调整，约 1-2 周。

**预期效果**：视觉理解能力质的飞跃，尤其对复杂室内场景（物体遮挡、小物体识别、文字标识）效果显著。

**参考**：[InternVL](https://arxiv.org/abs/2312.14238)，CVPR 2024。

---

### 16. Token 压缩替换：相似度合并 → Mamba 状态空间模型

**改进方案**：用 Mamba 的线性时间状态空间模型替代当前的滑动窗口 + 相似度合并机制，天然支持超长序列（数百步导航历史）。

**为什么难度高**：需要将 Mamba 集成为视觉历史的序列压缩器，与 LLM 的接口对齐，架构改动较大，约 2 周。

**预期效果**：注意力复杂度从 O(T²) 降为 O(T)，可处理极长导航轨迹（> 200 步）而不损失早期历史信息。

**参考**：[Mamba](https://arxiv.org/abs/2312.00752)，ICLR 2024。

---

### 17. 推理部署：TensorRT-LLM 引擎化

**改进方案**：将训练好的模型导出为 TensorRT-LLM 引擎，在 NVIDIA GPU 上实现极致推理性能。

**为什么难度高**：自定义多模态模型的 TRT-LLM 适配工作量大，需要手动编写 plugin，约 2 周。

**预期效果**：推理速度提升 5-10×，在 A100 上可实现 20+ Hz 在线推理（当前约 5 Hz）。

---

## 第六梯队：极高难度 × 极高效果（研究方向）

> 需要仿真环境、大量算力支持，适合作为后续发表方向。

---

### 18. 训练范式：SFT → PPO / GRPO 强化学习

**改进方案**：以导航成功率（SR）和路径效率（SPL）为奖励，用 PPO 或 GRPO 在 Habitat-Sim 仿真环境中在线强化学习。

**为什么难度极高**：

- 需要搭建 Habitat-Sim 在线仿真环境
- 需要设计奖励函数（稀疏 SR + 稠密距离奖励）
- 强化学习训练不稳定，需要大量 GPU 资源（通常 8-16 卡）

**预期效果**：导航成功率可突破 SFT 的天花板，在 VLN-CE R2R 等 benchmark 上有望进入 SOTA 行列；这是当前 LLM-based 导航最前沿的研究方向。

**参考**：EmbodiedGPT、NavRL 等工作。

### 改进 1 / 5 / 7：三种语言模型替换方案横向对比

三者均将 Vicuna-7B 替换为更强的语言模型，但优化目标、工程代价和适用场景各不相同。

| 维度 | 当前：Vicuna-7B | 改进 1：LLaMA-3.1-8B | 改进 5：Phi-3.5-mini | 改进 7：Qwen2.5-7B |
|---|---|---|---|---|
| **梯队** | — | 第一梯队（立刻可做） | 第二梯队（顺手可做） | 第三梯队（近期规划） |
| **难度** | — | 低 | 低 | 中 |
| **预期效果** | — | 高 | 中 | 高 |
| **参数量** | 7B | 8B | 3.8B | 7B |
| **MT-Bench** | 6.57 | 8.22 | 8.38 | 8.40 |
| **上下文长度** | 4K | 128K | 128K | 128K |
| **推理速度** | 基准 | 与基准相近 | 约快 2× | 与基准相近 |
| **中文能力** | 弱 | 无特别优势 | 无特别优势 | 极强（AlignBench 领先） |
| **架构兼容性** | — | `LlamaForCausalLM`，与现有完全一致 | Llama 架构变体，改动小 | `Qwen2ForCausalLM`，需新继承链 |
| **主要代码改动** | — | 换权重路径，升级 `transformers≥4.40` | 继承关系小改 | 改继承关系 + 新增对话模板 + 检查特殊 token |
| **核心优化目标** | — | 通用能力全面提升 | 速度 / 资源受限场景 | 中文指令理解 |

**各方案定位**

- **改进 1（LLaMA-3.1-8B）**：改动最小、收益最大的通用升级。架构与现有代码完全兼容，任何场景均可受益，应优先实施。
- **改进 5（Phi-3.5-mini）**：以参数量减半换取约 2× 推理速度，适合实机部署（在线导航频率从 5 Hz 提升至接近 10 Hz）或 GPU 显存受限的场景。MT-Bench 反而高于 Vicuna-7B，能力不降反升。
- **改进 7（Qwen2.5-7B）**：中文场景专项优化。若导航指令包含中文（如中文 R2R、ScanQA 中文版），理解准确率大幅提升；若任务全为英文，额外收益有限但工程代价更高（约一天适配工作量）。

**⚠️ 重要：三者互斥，必须三选一**

- ✅ **当前执行计划已选定改进 7**（后续任务可能包含中文指令）
- 若确认任务为纯英文、追求最小改动 → 可改选**改进 1**
- 追求实机部署 / 推理速度 → 选**改进 5**

| **输出维度** | 1408 | 1152 | 融合后需设计 | 3200 |

**⚠️ 改进 8 与 15 互斥，改进 13 只能叠加在改进 8 之上**

- 追求快速提升视觉理解 → 选**改进 8**
- 导航瓶颈是空间定位 → 在改进 8 基础上叠加**改进 13**
- 算力充足、追求极致视觉能力 → 选**改进 15**（长期规划）

| 维度 | 当前：固定余弦相似度阈值 | 改进 10：ToMe | 改进 12：可学习时序卷积 | 改进 16：Mamba |
|---|---|---|---|---|
| **难度** | — | 中 | 中 | 高 |
| **预期效果** | — | 中 | 中 | 高 |
| **重训成本** | — | **无需重训** | Stage 1+2（约 2-3 天） | Stage 1+2（约 2 周） |
| **改动位置** | — | EVA ViT Transformer Block 内 | `vlm_attention` 新增时序卷积模块 | 替换整个历史压缩器 |

| **动态场景表现** | 固定阈值，适应性差 | 软匹配，略优 | 端到端学习，最好 | 线性时间，理论最优 |
| **支持轨迹长度** | ~50 步 | ~50 步 | ~100 步 | >200 步 |
| **核心优化目标** | — | 免训练提速 | 动态场景理解 | 超长轨迹导航 |

**各方案定位**

- **改进 10（ToMe）**：无需重训，直接插入 ViT Block 即可使用，吞吐量提升约 2×，同时消除固定阈值的场景不适应问题。风险最低，应优先验证。
- **改进 12（可学习时序卷积）**：模型自动学习哪些帧重要，对快速移动、目标切换等动态场景更准确。需要 2-3 天完整训练，适合在改进 10 验证有效后进一步提升。
- **改进 16（Mamba）**：将历史建模复杂度从 O(T²) 降至 O(T)，可处理 200 步以上的超长导航轨迹而不丢失早期信息。工程难度较高，属于长期研究方向。

**⚠️ 三者互斥，按需选一个**

- 快速验证、无成本 → 选**改进 10**
- 动态场景导航质量不足 → 选**改进 12**
- 任务需要极长轨迹（>100 步） → 选**改进 16**

| **路径平滑度** | 一般（离散抖动） | 较好 | 极好 |

若最终目标是实机部署（改进 9），可直接跳过改进 4；若只做 benchmark 评测，改进 4 改动更小、风险更低。

- 只做 benchmark、改动小 → 选**改进 4**
- 面向实机部署、追求路径平滑 → 直接选**改进 9**

### 改进 6 / 2 / 18：三种训练范式方案横向对比

三者决定"用什么方法把模型教好"，对导航 SR 影响最直接。

| 维度 | 当前：纯 SFT | 改进 6：课程学习 SFT | 改进 2：SFT + DPO | 改进 18：PPO / GRPO |
|---|---|---|---|---|
| **难度** | — | 低 | 低 | 极高 |
| **预期效果** | — | 中（收敛加速） | 高（SR +3~8%） | 极高（突破 SFT 上限） |
| **重训成本** | — | 重跑 SFT | SFT 后继续 DPO（小时级） | 需仿真环境 + 多卡（周级） |
| **数据需求** | 正确轨迹 | 正确轨迹（按难度排序） | 成功/失败轨迹对 | 在线仿真交互 |
| **监督信号** | 正确动作标签 | 正确动作标签 | 成功 vs 失败的对比偏好 | SR / SPL 奖励函数 |
| **学到什么** | 正确答案是什么 | 正确答案是什么（学得更稳） | 为什么这个选择更好 | 怎么做才能真正到达目标 |
| **与其他关系** | — | 用于 SFT 阶段，可与改进 2 串联 | 在改进 6 之后执行 | 改进 2 的终极替代 |

**各方案定位**

- **改进 6（课程学习）**：只改变数据采样顺序（从易到难），无需修改模型结构，训练收敛速度加快 20-30%。可与改进 2 串联：先用课程学习跑 SFT，再做 DPO。
- **改进 2（SFT + DPO）**：在已有 SFT checkpoint 基础上继续用 DPO 做偏好对齐，让模型从"知道正确答案"升级为"知道为什么这个更好"。数据来源是仿真器日志中的成功/失败轨迹对，SR 预期提升 3-8%，是导航提升最确定的改进之一。
- **改进 18（PPO/GRPO）**：直接以导航成功率（SR）和路径效率（SPL）为奖励函数，在 Habitat-Sim 中在线强化学习，可突破 SFT / DPO 的天花板。需要搭建完整在线仿真环境 + 8-16 卡 GPU，属于长期研究方向。

**⚠️ 三者有串联顺序，改进 18 是改进 2 的终极替代**

- 推荐串联路径：**改进 6（SFT 阶段）→ 改进 2（DPO 对齐）→ 改进 18（有算力时）**
- 若直接做改进 18，改进 2 可跳过（两者目标重叠）
- 改进 6 对改进 2 / 18 均无冲突，可始终保留

### 改进 3 / 11 / 17：三种推理加速方案横向对比

三者**不提升导航能力（SR/SPL 不变）**，仅优化部署效率，应在导航能力改进验证完毕后再考虑。

| 维度 | 当前：HF generate + bf16 | 改进 3：AWQ 4-bit 量化 | 改进 11：vLLM | 改进 17：TensorRT-LLM |
|---|---|---|---|---|
| **难度** | — | 低 | 中 | 高 |
| **重训成本** | — | **无（量化现有 checkpoint）** | **无（换推理后端）** | **无（导出现有模型）** |
| **显存需求** | ~16 GB | ~8 GB | ~16 GB | ~16 GB（引擎化后降低） |
| **推理速度** | 基准 | 约 1.5× | 批量 3-5×（单步有限） | 5-10× |
| **多模态适配难度** | — | 低（只量化 LLM 部分） | 中（需注册视觉前处理） | 高（需手写 plugin） |
| **可与其他叠加** | — | 可与改进 11 叠加 | 可与改进 3 叠加 | 与改进 11 互斥 |
| **核心场景** | — | 消费级 GPU 部署 | 大规模 benchmark 评测 | 实机极致推理速度 |
| **对 SR/SPL 影响** | — | **零** | **零** | **零** |

**各方案定位**

- **改进 3（AWQ 量化）**：将模型权重从 bf16 压缩至 4-bit，显存从 16 GB 降至约 8 GB，可在 RTX 3090 / 4080 等消费级 GPU 上运行。精度损失 < 1%。改进 3 + 改进 11 可叠加（vLLM 原生支持 AWQ），是最实用的部署组合。
- **改进 11（vLLM）**：利用 PagedAttention 大幅提升批量推理吞吐，跑完整个 val_unseen 评测集的时间缩短 3-5×。对单步推理延迟改善有限，适合大规模 benchmark 跑分场景。
- **改进 17（TensorRT-LLM）**：最彻底的引擎化方案，推理速度提升 5-10×，A100 上可实现 20+ Hz 在线推理（当前约 5 Hz）。适配工程量大（约 2 周），与改进 11 互斥，选 17 则无需做 11。

**⚠️ 三者对导航能力无影响，不应占用导航改进的资源**

- 显存不足、需要在小卡上部署 → 选**改进 3**
- 需要快速跑完大量评测 → 选**改进 11**（可与改进 3 叠加）
- 面向实机实时导航、追求极致速度 → 选**改进 17**

| 11 | HF generate → vLLM | 批量评测吞吐提升 3-5× | 可与改进 3 叠加 |

> 改进 3 + 改进 11 可叠加，是最实用的组合；改进 17 是改进 11 的"终极替代"，选 17 则无需做 11。

**G. 独立模块（与其他改进均不冲突）**

| 改进 | 方案 | 特点 |
|---|---|---|
| 14 | MLP 投影头 → Resampler | 彻底解决 token 爆炸，与 LLM / 视觉编码器选择无关 |

> 改进 14 是独立模块，可与 A~F 任意组合并联实施。

#### 改进方案兼容性与资源成本总览

跨模块之间**均可并联**，但不同模块的重训成本差异悬殊，**选方案时必须将重训成本纳入决策**，避免资源浪费。

| 模块 | 候选改进 | 选择约束 | 重训成本 | 训练内容 | 能否立即并联？ |
|---|---|---|---|---|---|
| **A 语言模型** | 1 / 5 / 7 | 三选一 | Stage 1 + Stage 2 | projector 对齐 + 全量微调 | 需提前选定，不可中途切换 |
| **B 视觉编码器** | 8 / 15（+13 可叠） | 8 或 15 二选一 | 8：仅 Stage 1（约 1-2 小时）；15：Stage 1+2（约 1-2 周） | projector 重新对齐 | 改进 8 成本低，可快速并联 |
| **C Token 压缩** | 10 / 12 / 16 | 三选一 | 10：**无需重训**；12/16：Stage 1+2（数天） | 10 直接插入 ViT Block | 改进 10 零成本，优先并联 |
| **D 动作输出** | 4 / 9 | 二选一 | Stage 1 + Stage 2 | 输出格式改变，需重新学习 | 与目标场景强绑定，成本高 |
| **E 训练范式** | 6 → 2 → 18 | 串联，可止于任意阶段 | 6：重跑 SFT；2：DPO 继续训练（小时级）；18：在线 RL（极高） | 18 需仿真环境 + 多卡 | 6 和 2 可并联，18 需专项资源 |
| **F 推理加速** | 3+11 可叠 / 17 替代 11 | 3+11 或 17 | **无需重训**（量化/换后端，分钟级） | checkpoint 不变 | 随时并联，成本可忽略 |
| **G 投影头** | 14 | 独立，无冲突 | Stage 1 + Stage 2（约 1-2 周） | Resampler 结构改动，周期长 | 独立推进但成本高，列入长期规划 |

**重训成本等级一览：**

```
无需重训   ── 模块 F（改进 3、11、17）、模块 C 改进 10（ToMe）
仅 Stage 1 ── 模块 B 改进 8（SigLIP，约 1-2 小时）、改进 13（DINOv2）
Stage 1+2  ── 模块 A（改进 1/5/7）、模块 C（改进 12/16）、模块 D、模块 E（改进 6/2）
极高成本   ── 模块 B 改进 15（InternViT-6B）、模块 E 改进 18（PPO/GRPO）、模块 G 改进 14
```

**选方案前的三个关键决策：**

1. **首要：选定模块 A（语言模型基座）**
   已选定改进 7（Qwen2.5-7B），兼顾中英文场景。若后续确认任务为纯英文，可评估切换回改进 1，但 Stage 1 需重训。

2. **同步并联无成本改进（模块 F + 改进 10）**
   改进 3（AWQ）、10（ToMe）、11（vLLM）无需重训，可在模块 A 训练期间同步推进，不额外占用 GPU 资源。

3. **按资源量力排队，低成本验证优先**
   成本高的改进（模块 C 改进 12/16、模块 G 改进 14、模块 B 改进 15）应在低成本改进验证有效后再投入，避免因效果不及预期浪费长时间训练资源。

**推荐启动顺序（资源受限场景）：**

```
第一批（无成本，立即可做）
  └─ 模块 F：改进 3（AWQ 量化）+ 改进 11（vLLM）
     模块 C：改进 10（ToMe，免训练，直接插入 ViT）

第二批（低成本，1-2 天内出结果）
  ├─ 模块 A：已选定改进 7（Qwen2.5-7B），启动训练（占 GPU，后台跑）
  └─ 模块 B：改进 8（SigLIP，仅 Stage 1，约 1-2 小时）

第三批（中等成本，基于前两批效果决定是否继续）
  ├─ 模块 E：改进 6（课程学习）→ 改进 2（DPO）
  └─ 模块 C：改进 10 效果不足时，考虑改进 12（可学习时序卷积）

长期规划（高成本，需专项资源，择机投入）
  ├─ 模块 B：改进 15（InternViT-6B，约 1-2 周）
  ├─ 模块 G：改进 14（Resampler 投影头，约 1-2 周）
  └─ 模块 E：改进 18（PPO/GRPO 强化学习，需仿真环境 + 多卡）
```
